**Assigned agent**: `divinr-test-agent` (all four).

**Suspected shared root cause**: stored auth state stale. `apps/e2e/.auth/testing-team.json`
was captured 2026-04-19T20:44; the cron pass fired 2026-04-20T06:44 (~10 hours
later). Supabase access tokens default to a short TTL. Three of the four specs
(portfolios, performance, authoring) land on the target page but never paint
either a populated row or a zero-state — consistent with a silently-401'd data
fetch. The fourth (admin) surfaces an explicit 401 from a raw
`page.request.get('/api/billing/subscription')`.

**Recommended fix ordering for divinr-test-agent**:

1. Regenerate auth state via `apps/e2e/scripts/prepare-auth-state.ts` and
   re-run all four facet projects. If all four go green, the root cause is
   confirmed and the fix is "refresh auth before each cron pass" — either by
   prepending the auth-state script to the discover-cron wrapper, or by
   reducing the stored state's usable window.
2. If the admin spec still 401s after a fresh auth state, the secondary root
   cause documented in the `dd97ef65` finding applies: Playwright's
   `APIRequestContext` (`page.request.get`) bypasses the app's Authorization
   header wrapper. Rewrite the spec to either (a) evaluate the bearer
   fetch inside the page context, or (b) read the stored token from the
   storage state and attach it explicitly to the request.
3. If portfolios/performance/authoring still fail after a fresh auth state,
   escalate to `divinr-product-agent` — at that point the issue is a real
   rendering regression, not an auth artifact.
